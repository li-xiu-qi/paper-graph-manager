# Paper Graph Manager 技术选型策略

> 本文档记录项目关键的技术选型决策及其原因，供后续迭代参考。

---

## 1. 核心定位

**关键词/正则优先，轻量本地优先，不做向量化。**

本项目定位为个人论文管理工具，核心场景是：
- 小规模个人论文库（几十到几百篇）
- 本地优先、离线可用
- 交互式图谱探索 + 轻量智能标注

因此技术选型围绕**简单、可控、低运维**展开，避免引入重型基础设施。

---

## 2. 知识图谱与 LLM 结合策略

### 2.1 图谱构建：规则驱动，LLM 增强数据

| 层面 | 实现方式 | 原因 |
|------|----------|------|
| 图谱结构 | 纯规则（作者共现、论文共著） | 确定性高、可解释、无额外成本 |
| 图谱数据丰富度 | LLM 标注（core_contribution、teams） | 规则无法提炼语义信息 |

**不采用 LLM 做图谱构建的原因：**
- 个人库规模小，规则足够覆盖显式关系
- LLM 生成的关系需要人工校验，增加心智负担
- 本地运行 LLM 成本高，云端调用延迟大

### 2.2 聊天检索：关键词 + 正则，不用 Embedding

**当前实现：**
- 结构化查询：关键词匹配 `title` / `abstract`
- 搜索意图：正则提取查询词
- RAG 模式：abstract 前 20 字符子串匹配

**明确不使用 Embedding 的原因：**
1. **定位不符**：个人工具不需要语义搜索的"模糊匹配"能力，用户更习惯精确控制
2. **运维成本**：向量数据库增加部署复杂度，与"本地轻量"定位冲突
3. **数据规模**：个人库通常 < 1000 篇，关键词匹配召回率足够
4. **可解释性**：正则/关键词匹配结果可预测，用户能理解为什么匹配

**未来 reconsider 的条件：**
- 论文库超过 5000 篇且关键词匹配准确率明显下降
- 用户明确需要"语义相似"检索场景
- 本地已部署 embedding 模型（如 Ollama + nomic-embed-text）

### 2.3 图谱问答（Graph QA）

**当前实现**：字符串匹配 + 规则路由

**不采用 LLM 图谱遍历的原因：**
- 图谱节点/边数量有限，规则查询足够
- LLM 做图谱推理需要 prompt engineering，收益不确定
- 保持系统透明：用户看到的是数据库记录，不是 LLM 推断

**可能的增强方向（非当前优先级）：**
- 图谱统计指标（中心度、社区）用 LLM 生成自然语言摘要
- 节点点击后 LLM 生成该节点的智能摘要（需要额外 prompt）

## 3. 数据存储

### 3.1 数据库：SQLite

**选择原因：**
- 本地优先，无需独立数据库服务
- 单文件备份/迁移方便
- 支持 pandas 直接查询，数据分析友好
- 并发场景简单（单用户个人工具）

**不选 PostgreSQL/MySQL 的原因：**
- 运维成本过高，与个人工具定位不符
- 个人库规模下 SQLite 性能完全足够

### 3.2 图存储：NetworkX（内存计算）

**选择原因：**
- 轻量，无额外依赖
- 与 pandas 生态兼容
- 适合 < 1000 节点的个人图谱

**不选 Neo4j 的原因：**
- 需要独立服务，运维成本高
- 个人图谱规模下 NetworkX 性能足够

---

## 4. 前端技术

### 4.1 图可视化：PixiJS + 自定义力导向

**选择原因：**
- 需要交互式探索（缩放、平移、点击节点）
- 项目已有 PixiJS 技术积累
- 比 D3.js 更适合大规模节点渲染

**不选 Cytoscape.js / Sigma.js 的原因：**
- 项目技术栈已确定 PixiJS
- 自定义渲染更灵活，可深度集成业务逻辑

### 4.2 UI 框架：shadcn/ui + Tailwind CSS

**选择原因：**
- 组件库与 Tailwind 生态成熟
- 项目已有使用基础
- 快速迭代，适合个人项目

---

## 5. 标注与模型

### 5.1 LLM 调用：OpenAI 兼容接口

**选择原因：**
- Step Plan / OpenAI / 本地 Ollama 均兼容
- 不绑定单一厂商
- 支持 JSON mode（结构化输出）

**当前配置：**
- 默认模型：`step-3.7-flash`
- 默认 Base URL：`https://api.stepfun.com/step_plan/v1`
- 可通过 `.env` 灵活切换

### 5.2 标注 Prompt 设计原则

1. **结构化输出**：强制 JSON 格式，便于入库
2. **低温采样**：`temperature=0.3`，保证稳定性
3. **截断控制**：abstract 限制 3000 字符，控制 token 消耗
4. **后处理清洗**：去除 markdown 代码块标记，容错 LLM 输出格式

## 6. 测试策略

### 6.1 后端：单元测试 + API 集成测试

- 核心模块（annotate、notes、graph、ingest）单元测试覆盖
- FastAPI 端点用 `TestClient` 做集成测试
- 覆盖率目标：> 90%（当前已达成）

### 6.2 前端：API 层测试 + Smoke Test

- `services/api.test.ts`：所有 API 函数的成功/失败路径
- `pages.test.tsx`：页面组件可渲染性验证
- **不引入组件交互测试**：项目规模小，手动测试成本低

### 6.3 E2E：Python 集成脚本

- `test_e2e.py`：覆盖数据库初始化、入库、标注、图谱、笔记全流程
- 不引入 Playwright/Cypress：个人项目迭代快，E2E 维护成本高

---

## 7. 明确不采用的技术

| 技术 | 原因 |
|------|------|
| Embedding / 向量数据库 | 定位不符，个人库规模不需要，运维成本高 |
| Neo4j / 图数据库 | 个人图谱规模小，NetworkX 足够 |
| PostgreSQL / MySQL | 单用户本地场景，SQLite 足够 |
| Playwright / Cypress | 个人项目，手动测试 + 单元测试足够 |
| LangChain / LlamaIndex | 场景简单，直接调用 OpenAI API 更可控 |
| 微服务架构 | 个人项目，单体 FastAPI 足够 |
| Docker 化 | 本地直接运行，增加复杂度 |

---

## 8. 技术债务与未来 reconsider

1. **PDF 解析**：当前用 PyMuPDF，后续如果 PDF 格式复杂可考虑 `pdfplumber` 或 `camelot`
2. **图谱布局**：当前 PyVis 力导向布局，节点多时可考虑预计算布局（如 fa2）
3. **并发安全**：当前 SQLite 无并发控制，多设备同步需求出现时需迁移
4. **备份策略**：当前无自动备份，数据量大时需考虑定期导出
